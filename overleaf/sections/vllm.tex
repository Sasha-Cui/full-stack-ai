\section{vLLM and Efficient LLM Serving}
\label{sec:vllm}

\subsection{Motivation}
\label{sec:vllm:motivation}

Modern LLM systems are often bottlenecked not by raw model quality but by serving throughput, latency, and memory efficiency. vLLM addresses this systems problem directly. Its goal is to keep GPUs busy while serving many requests with different prompt lengths and different generation horizons. This makes it useful in both deployment settings and research workflows that require large-scale sampling, such as evaluation, synthetic data generation, and reinforcement learning rollouts.

The most important idea behind vLLM is that efficient decoding requires both:
\begin{itemize}
  \item \textbf{fast reuse of past computation}, via the key-value cache; and
  \item \textbf{good memory management}, so those caches can coexist for many concurrent requests.
\end{itemize}

\subsection{Why the KV Cache Matters}
\label{sec:vllm:kv}

The standard scaled dot-product attention operation is
\[
\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V,
\]
where $Q$, $K$, and $V$ denote the query, key, and value tensors. During autoregressive decoding, each new token must attend to all previous tokens in the sequence. Without caching, this means repeatedly recomputing keys and values for the same prefix.

The key-value (KV) cache stores those previously computed keys and values so that each decoding step only computes new entries for the newest token. This changes the practical serving story:
\begin{itemize}
  \item \textbf{Without a KV cache:} prefixes are recomputed repeatedly, wasting compute.
  \item \textbf{With a KV cache:} the model reuses prior work and spends compute on the new token only.
\end{itemize}

\begin{center}
\small
\begin{tabular}{p{0.28\linewidth} p{0.30\linewidth} p{0.27\linewidth}}
\toprule
Setting & What gets recomputed & Practical consequence \\
\midrule
No KV cache & Keys and values for the whole prefix & High latency and poor throughput on long contexts \\
KV cache enabled & Only the newest keys and values & Much faster incremental decoding \\
\bottomrule
\end{tabular}
\end{center}

Caching solves the compute problem, but it creates a memory problem: every live request now owns a growing cache in GPU memory.

\subsection{The Real Serving Problem: Many Requests at Once}
\label{sec:vllm:serving}

Serving one long sequence is relatively straightforward. Serving thousands of requests is not. Real workloads have three characteristics that make naive implementations inefficient:
\begin{itemize}
  \item \textbf{Asynchronous arrivals:} requests appear at different times.
  \item \textbf{Variable lengths:} prompts and completions have different sizes.
  \item \textbf{Uneven completion times:} some requests finish immediately while others continue decoding.
\end{itemize}

These properties make static batching inefficient and contiguous memory allocation fragile. A practical serving runtime must handle both scheduling and memory layout at the same time.

\subsection{PagedAttention: The Core vLLM Insight}
\label{sec:vllm:paged}

The main systems contribution of vLLM is PagedAttention~\cite{kwon2023efficient}. Instead of storing each request's KV cache in one large contiguous buffer, vLLM splits the cache into fixed-size pages. A request then owns a list of pages rather than one monolithic block.

This indirection is powerful because it avoids the usual memory-fragmentation problem:
\begin{itemize}
  \item finished requests can release pages immediately;
  \item long-running requests can acquire new pages without large copies;
  \item the runtime can repack work implicitly through page allocation rather than through expensive memory movement.
\end{itemize}

Conceptually, the runtime maintains a page table that maps logical token positions to physical memory pages. Attention kernels can then read KV values in logical order even if the underlying storage is physically scattered.

\subsection{Continuous Batching and Scheduling}
\label{sec:vllm:scheduling}

Memory efficiency alone is not enough; the scheduler must also keep the GPU occupied. vLLM therefore uses continuous batching. Instead of waiting for a fixed batch to finish, it repeatedly mixes:
\begin{itemize}
  \item new requests entering the system,
  \item existing requests that still need another token, and
  \item requests that can be removed because they have reached a stop condition.
\end{itemize}

This approach improves utilization because the active batch can change at every decoding iteration. It is especially helpful for bursty real-world workloads and for research jobs that mix short prompts with long completions.

\subsection{Why Researchers Care}
\label{sec:vllm:research}

For full-stack researchers, vLLM is useful in at least four recurring settings:
\begin{itemize}
  \item \textbf{Benchmarking:} evaluating many prompts quickly.
  \item \textbf{Synthetic data generation:} sampling large corpora for SFT or preference data.
  \item \textbf{Agent rollouts:} generating many trajectories for tool-using systems.
  \item \textbf{RL pipelines:} producing rollouts faster so the trainer is not starved for experience.
\end{itemize}

This is why the repository pairs vLLM material with DeepSpeed and VERL. vLLM handles the inference-heavy side of the loop, while the training stack handles the gradient-heavy side.

\subsection{Minimal Example}
\label{sec:vllm:demo}

\subsubsection{Installation}
\begin{verbatim}
pip install -U vllm transformers accelerate
\end{verbatim}

\subsubsection{Basic Usage}
\begin{verbatim}
from vllm import LLM, SamplingParams
import torch

if torch.cuda.is_available():
    print(f"GPU: {torch.cuda.get_device_name(0)}")
else:
    print("No GPU found, vLLM will be limited")

llm = LLM(
    model="Qwen/Qwen2-1.5B-Instruct",
    trust_remote_code=True,
)

params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=128,
)

prompts = [
    "Tell me a short story about a robot.",
    "List three benefits of paged KV caching."
]

outputs = llm.generate(prompts, params)

for i, output in enumerate(outputs):
    print(f"Response {i+1}:")
    print(output.outputs[0].text)
\end{verbatim}

The accompanying \texttt{MWEs/vllm+deepspeed/} notebooks go further and place vLLM in the broader serving and post-training context.

\subsection{Summary}
\label{sec:vllm:summary}

vLLM matters because efficient inference is a systems problem, not just a model problem. KV caching reduces redundant computation, PagedAttention reduces memory fragmentation, and continuous batching keeps the hardware busy. Together, these ideas make large-scale LLM serving and rollout generation practical.
