\section{DeepSpeed and Memory-Efficient Training}
\label{sec:deepspeed}

\subsection{Motivation: The Training Memory Wall}
\label{sec:deepspeed:motivation}

Large-model training is often limited by memory before it is limited by raw compute. A model that fits comfortably for inference may still be impossible to train because the training loop must also store gradients, optimizer states, activations, and temporary communication buffers.

\begin{center}
\small
\begin{tabular}{p{0.24\linewidth} p{0.34\linewidth} p{0.27\linewidth}}
\toprule
Component & Role during training & Memory intuition \\
\midrule
Parameters & Current model weights & About one model copy \\
Activations & Saved tensors for backpropagation & Scales with depth, batch size, and sequence length \\
Gradients & Per-parameter derivatives & Roughly one model copy \\
Optimizer states & Momentum and variance terms for Adam-like optimizers & Often two extra copies \\
Temporary buffers & Workspace for kernels and communication & Variable overhead \\
\bottomrule
\end{tabular}
\end{center}

For Adam-style optimizers, the optimizer states alone can dominate the memory budget. This is why naive data parallelism does not solve the real problem: each worker still stores nearly all training states locally.

\subsection{Why Naive Data Parallelism Is Wasteful}
\label{sec:deepspeed:dp}

In ordinary data parallel training, every GPU keeps a full replica of:
\begin{itemize}
  \item the model parameters,
  \item the gradients, and
  \item the optimizer states.
\end{itemize}

This is easy to reason about and straightforward to implement, but it is memory-inefficient. Adding more GPUs increases aggregate cluster memory, yet it does not proportionally reduce the amount each individual GPU must hold. As a result, some models remain impossible to train even when plenty of total cluster memory exists.

\subsection{The ZeRO Idea}
\label{sec:deepspeed:zero}

DeepSpeed's central idea is ZeRO: the Zero Redundancy Optimizer. Instead of replicating all training states on every device, ZeRO shards those states across the workers and reconstructs what is needed during computation. The high-level principle is simple:
\begin{quote}
Do not store the same training state on every GPU unless the algorithm truly requires it.
\end{quote}

This idea is powerful because most optimizer states are naturally partitionable. Consider Adam's elementwise update:
\begin{align}
  m_i^t &\leftarrow \beta_1 m_i^{t-1} + (1 - \beta_1) g_i^t, \\
  v_i^t &\leftarrow \beta_2 v_i^{t-1} + (1 - \beta_2) (g_i^t)^2, \\
  \theta_i^t &\leftarrow \theta_i^{t-1}
  - \alpha \frac{m_i^t / (1 - \beta_1^t)}
  {\sqrt{v_i^t / (1 - \beta_2^t)} + \epsilon}.
\end{align}

Each parameter $\theta_i$ only needs its local optimizer states and its own gradient. That means the optimizer states can be sharded naturally across devices.

\subsection{ZeRO Stages}
\label{sec:deepspeed:stages}

\paragraph{ZeRO-1: shard optimizer states.}
Each worker still keeps full parameters and gradients, but only stores optimizer states for a subset of parameters. This removes a large fraction of the memory overhead while requiring relatively modest communication.

\paragraph{ZeRO-2: shard optimizer states and gradients.}
In naive data parallelism, every worker ends backpropagation with the full gradient tensor. ZeRO-2 replaces the usual all-reduce pattern with reduce-scatter style communication so that the reduced gradients remain partitioned.
\begin{itemize}
  \item \texttt{all-reduce}: every worker receives the full reduced gradient.
  \item \texttt{reduce-scatter}: the reduction is performed, but the result stays sharded.
\end{itemize}

\paragraph{ZeRO-3: shard parameters as well.}
The final stage also shards the model parameters themselves. This provides the largest memory savings, but it also introduces the most communication because parameters must be gathered when layers are used and then released again.

\begin{center}
\small
\begin{tabular}{p{0.14\linewidth} p{0.38\linewidth} p{0.24\linewidth}}
\toprule
Stage & What is sharded? & Main effect \\
\midrule
Naive DP & Nothing & Simple, but memory-heavy \\
ZeRO-1 & Optimizer states & First major memory savings \\
ZeRO-2 & Optimizer states and gradients & Lower backprop footprint \\
ZeRO-3 & Parameters, gradients, and optimizer states & Maximum memory savings \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Why This Matters in Practice}
\label{sec:deepspeed:practice}

DeepSpeed is useful whenever model size is constrained by training-state memory rather than by pure FLOPs. This is common in:
\begin{itemize}
  \item large supervised finetuning runs,
  \item RL post-training where policy updates are expensive,
  \item experiments on long contexts or large batch sizes, and
  \item multi-node settings where memory efficiency determines whether a run is feasible at all.
\end{itemize}

The repository includes a DeepSpeed example in the Ray training folder and a paired serving/training notebook collection in the vLLM+DeepSpeed folder. These materials should therefore be viewed less as isolated demos and more as building blocks for realistic large-model training workflows.

\subsection{Summary}
\label{sec:deepspeed:summary}

DeepSpeed's importance comes from a single systems insight: most of the training memory budget is redundant across workers. ZeRO removes that redundancy step by step. For full-stack AI researchers, understanding this design is essential because many practical training decisions boil down to a tradeoff between memory savings, communication overhead, and implementation complexity.
